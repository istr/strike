# ADR-018: Cryptographic Test Material Is Ephemeral

## Status

Accepted.

## Context

Test suites for cryptographic operations need keys, certificates,
and signed payloads. The default approach in many projects is to
commit a "test PKI" tree to the repository: a CA cert, a few server
certs, a few keys, all under `testdata/`. This makes tests
self-contained and reproducible across CI runs.

It also creates four problems that are individually small and
collectively serious:

- *Committed key material drifts toward production-shaped*. A test
  CA used for years acquires the trappings of a production CA --
  longer validity, more SANs, conventional naming. Operators
  copy-paste from it. The line between "test fixture" and "convenient
  cert I have lying around" gets blurred.
- *Discovered keys cannot be revoked from history*. A test private
  key committed to git is recoverable from git history forever.
  If a copy-paste later promotes that key to a real role, history
  is the leak.
- *Test runs are not isolated*. Two CI runs share the same key,
  the same serial numbers, the same timestamps. A bug that depends
  on key reuse passes in one run and fails in another only by
  coincidence.
- *The "no committed secrets" linting rule has a permanent
  exception*. Every secret-scanner has to be configured to ignore
  the testdata directory, which means the rule is "no committed
  secrets except where there are committed secrets". The rule is
  weaker than its phrasing.

The strict alternative is to generate every piece of cryptographic
test material at test time, in memory, and never write it to disk
beyond what `t.TempDir()` cleans up. The cost is a few hundred
microseconds per test (an ECDSA P-256 keygen and a self-signed
cert). The benefit is a fully enforceable rule.

## Decision

All cryptographic test material in strike is ephemeral, generated
at test time via the Go standard library, and never committed to
the repository.

The TLS test helpers `newTLSTestEngine` and `newMTLSTestEngine` in
`internal/container/testpki_test.go` generate a fresh CA and server
cert per test (and a client cert for mTLS tests). Signing tests
generate a fresh `*ecdsa.PrivateKey` per test using
`ecdsa.GenerateKey(elliptic.P256(), rand.Reader)`. There is no
`testdata/keys/` directory; there is no `testdata/certs/` directory;
there is no shared "test CA" reused across tests.

The repository's `.gitignore` includes `*.pem`, `*.key`, and
`cosign.*` to prevent accidental commits. Pre-commit hooks and
secret scanners apply uniformly across the tree, with no testdata
exception.

Cross-validation vectors (per ADR-017) follow the same rule:
vectors that involve signing operations have a `key_pem` field
that is *parametrized*, not populated. The Go test suite injects
an ephemeral key when running the vector; a second implementation
does the same. The vector's `expected` field captures values
derivable from the ephemeral key (e.g. the DER-encoded public key
in base64) but no values that would require the private key to be
shared.

Integration tests against a real container engine generate keys
the same way. A test that needs to sign an OCI image creates a key
in the test, signs the image, and verifies the signature within
the same test invocation. The key never leaves the test process.

## Consequences

- The repository has a complete and enforceable "no committed
  secrets" rule, with no testdata exception. Secret scanners can
  run in default-strict mode.
- Test runs are isolated by construction. A flaky test that depends
  on key reuse fails consistently rather than intermittently,
  because no two runs share a key.
- Cross-validation vectors stay portable across implementations
  without smuggling a Go-specific test key into a Rust test suite.
  The key is always generated by the implementation under test;
  the vector specifies what the test must do with it.
- The cost is a per-test ECDSA keygen, which is in the order of
  hundreds of microseconds. For test suites with thousands of
  tests, this adds a noticeable but bounded overhead. In strike's
  current scale, the cost is invisible.
- A new contributor cannot bypass the rule by "just committing the
  test cert temporarily". The pattern of inline generation is
  visible in every existing test; a PR that introduces a committed
  cert stands out in review.

## Principles

- Secrets are typed (extended to test material: ephemeral
  generation is the operational form of "secrets do not exist on
  disk")
- Code is liability (no shared test-PKI infrastructure to maintain,
  no expiry to manage, no rotation to schedule)
- External references are digest-pinned (test outputs are content-
  addressed, including signatures whose payload is reproducible
  from the inputs and the ephemeral key)
