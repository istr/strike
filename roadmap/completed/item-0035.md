Motivation: today the cue toolchain is installed twice -- the CLI binary and the cuelang.org/go Go API via go.mod -- and the two must be kept in sync by hand. The CLI is invoked only from the Makefile (specs, generate, lint-cue-fmt targets), which the project wants to retire in favour of a single go.mod source of truth.

Research finding (analysis, against cuelang.org/go at the go.mod-pinned commit): two of the three CLI uses are trivial public-API reimplementations; only gengotypes is hard.
- cue export (#Lane, #Attestation, trustlayers JSON fixtures): pure value evaluation, fully covered by public cue/load plus cue.Value.MarshalJSON.
- cue fmt --check --files specs: covered by public cue/format (format.Node / format.Source) over the parsed specs files.
- cue exp gengotypes: the generator lives in cuelang.org/go/internal/encoding/gotypes, under internal/, so it cannot be imported across the module boundary. It is a single ~850-line file whose only non-stdlib imports are public CUE packages plus golang.org/x/tools astutil, but vendoring it would re-introduce a source-copy sync burden and ~850 lines of liability, and it is explicitly an experimental command. Reimplementing it loses the hard-won exact semantics (the @go(-) disjunction handling, cross-package @go target emission).

Ratified direction: keep the cue CLI but as a go.mod tool dependency (cuelang.org/go/cmd/cue), so MVS unifies it with the library version and the sync is automatic; gengotypes runs via that tool, go generate driven, not a separate install. export and fmt-check become small internal Go tools using only public CUE APIs. CUE requires go 1.25, so the go 1.24+ tool directive is available; cmd/cue is a normal package main; tool deps do not enter the shipped controller binary.

No-exec note: this is build-time codegen, outside the runtime-controller no-exec principle, which binds only the attested runtime path. Spawning cue at build time does not touch that surface.

Design fork to resolve at execution (drift-invariant, not byte-exact): where the export/fmt internal tools live. tools/lintfrom sets the precedent of a separate per-tool go.mod, which keeps tool-only deps out of the main graph but re-creates a second cuelang.org/go to track; the alternative is a main-module command sharing the single pinned version. The gengotypes go tool dependency must live in the main go.mod for the MVS unification to hold. Decide the export/fmt placement against the code-is-liability and single-source goals at execution.

Spike before any byte-exact instruction: confirm go tool cue exp gengotypes runs headless from the tool directive and produces byte-identical output to the current CLI invocation, and that MVS unifies cmd/cue with the library version. Generator-internal behaviour: measure in a throwaway worktree, do not guess.

An ADR recording the go-tool-over-CLI build-toolchain decision may be warranted; author it append-only at execution time, never in this store. The byte-exact instruction (exact Makefile edits, go:generate directives, tool file contents) is authored ephemerally against the then-current pin and discarded after landing.

Sequencing: the export targets operate on the post-item-0032 CUE-first specs; this can proceed independently of the spec-layering arc but should be validated against whatever spec file set is current at execution.

## Final summary

Landed at 1fc9bb1. cuelang.org/go/cmd/cue added as a go.mod tool dependency (MVS-unified at v0.17.0); every Makefile cue invocation replaced by go tool cue, hand-installed binary retired.
