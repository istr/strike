Drift-invariant notes:

This is the culmination of the spec-layering arc, which began as a critique of
the CUE structure. Earlier items cleaned the scalar typing (item-0032) and the
Go-side type surface (item-0034); this item fixes the structural root: the
mapping from CUE packages to Go packages.

Diagnosis (ratified 2026-06-25): almost everything lives in CUE package lane and
is generated into internal/lane (services tier), whether it is foundational
vocabulary (base-scalars, base-transport, base-peer, base-provenance,
base-target) or a high-tier wire/api consumer (api-*, wire-*). Because vocabulary
and consumer share one package, references are intra-package -- no import, @go()
targets intact -- which is exactly why it "just works" today and exactly why the
vocabulary is pinned to a high tier. That single co-location is the structural
root: it forces the transport -> lane direction (lane's peer/wire types embed
transport vocabulary while transport sits below lane) and every cross-tier string
boundary item-0034 has to hoist (transport cannot reference lane.Identifier). The
single-package-lane export is legacy, pre-clean-up, not a chosen design.

Invariant this item establishes: a CUE package maps to one Go package at the
layer its content belongs to, and the CUE import DAG is the arch-lint tier DAG --
vocabulary at the bottom (foundation), consumers above, dependencies downward
only. Once that holds, the CUE setup is close to final: the mapping is principled
and stable rather than a historical accident, and the cross-tier cycles cannot
recur by construction.

Spike result (spike-0036-s1, cue v0.17.0-rc.1, anchor 8418503): the load-bearing
unknown is RESOLVED, and the answer redirects the design.

- The ADR-047 (3) concern -- "adding an import to a CUE file makes that file's
  @go() target be ignored" -- did NOT reproduce. In both the failing and the
  passing run the field-level @go() targets were preserved identically to the
  no-import control (TheToken/Note/Holder). Target handling is not the blocker.
- The real constraint is structural and layout-bound: gengotypes binds the output
  Go file to the INPUT CUE package's directory ("code is generated in each CUE
  package directory at cue_types_${pkgname}_gen.go") and transitively regenerates
  imported deps into that same directory, while Go forbids two packages in one
  directory. In strike's FLAT specs/ layout every CUE package shares one
  directory, so an importer plus its dependency collide: rc=1, "cannot generate
  two Go packages in one directory". --outfile does not decouple it (it redirects
  one package; the importer drags its dep along).
- With ONE CUE package per directory the importer generates cleanly (rc=0), the
  cross-package Go import is real (import ".../vocab", field typed vocab.Token),
  the output compiles (go build rc=0), and the @go() targets are preserved.

So the generated restructure IS viable, conditional on giving each CUE package
its own directory. The flat specs/*.cue layout and the filename prefix are what
must go.

Mechanical consequences the design session must absorb (flagged, not designed
here):
- Per-directory layout: each CUE package (the new vocabulary, api, wire, and the
  existing attest/crossval/trustlayers) moves into its own subdirectory under
  specs/.
- Generated filename: addressed by directory, the package name is implied by the
  import path, so the file is cue_types_gen.go without the _<pkg>_ infix -- the
  current cue_types_lane_gen.go convention changes.
- Generation flow: the Makefile's "cue exp gengotypes ./specs:lane then mv
  specs/cue_types_lane_gen.go into internal/lane/" assumes the flat layout and a
  single package. Under per-directory packages each package generates its own
  cue_types_gen.go in its own specs subdirectory, and each must be routed to its
  Go home under internal/. How the specs subdirectory import paths map onto the
  internal/* Go package locations (the generate-then-route step) is the central
  generation-plumbing question for the design.

Target structure (refine exact package names and Go homes at design time): the
base-* vocabulary becomes its own CUE package in its own directory, generating
into a foundation-tier Go package (the natural home for the contract's
vocabulary, depending only on the contract layer); transport, lane, mediator, and
the rest import it downward; lane keeps its DAG/State logic and the api/wire
generated types but sources the scalar and structural vocabulary cross-package.
attest, crossval, and trustlayers are already separate CUE packages and are
confirmed at their correct layers; under the per-directory rule they also each
get their own directory.

Relationship to other items:
- Supersedes item-0033: the transport CUE package is one slice of this carve-out,
  not a separate move. Set item-0033 to superseded (executor lane).
- Unblocks item-0034's cross-package dimension: after this lands, the only
  remaining type-cleanliness work is the intra-lane Identifier/DAG/State cascade
  and the per-family widenings, all free of cycle-forced boundaries. Sequence

ADR-047 is revisited deliberately: (3) kept the generated layers in one CUE
package to avoid an @go()-on-import risk that the spike has now shown does not
exist; the actual layering cost is the per-directory output rule, which is
acceptable. The amendment is an append-only edit made at design/execution time,
recording both the empirical import result and the per-directory layout it
mandates -- never written here. The byte-exact instructions and the structural
design are authored with fresh context against the then-current pin; nothing
byte-exact is stored here.
