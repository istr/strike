Closes the measured duplication between the two Go-type linters (comparison A of
the tooling review): a shared strike-named-type resolver, a shared allowlist
idiom, and a shared package loader, duplicated across two modules. More deeply,
linttypeconv's conversion-ownership check is a subset of the conversion facts the
linttypeflow extractor already emits, so it folds in as an ownership gate over
existing facts rather than a second detector.

golangci-lint constraint (hard): linttypeconv is today the go/analysis analyzer
form (golangci-consumable via a custom plugin); linttypeflow is a raw go/packages
CLI. The merge must preserve an analyzer entrypoint, not collapse into a
make-only CLI. Dual-run is the target -- golangci-lint plugin and standalone
binary, both go-native, neither needing make or a shell. That is the same
beta-readiness direction the README first sentence implies ("no shell, no local
toolchain"): keep the tools go-native so the Makefile wrappers can fall away when
make is retired (item-0035 and the beta-readiness arc), rather than adding a
second orchestration layer.

Execution design fork (drift-invariant, resolve at execution): reconcile
linttypeflow's -report JSONL survey (all fact kinds, standalone, exit zero) with
the go/analysis diagnostic model (per-finding gates). The likely shape is
analyzer(s) for the gates plus a standalone -report entrypoint reusing the one
extractor; decide the exact split against the code at execution.

Sequencing: depends on linttypeflow (item-0073). Best sequenced after the
escalation and standalone-to-aggregate graduation (item-0084) settle the gate
surface, so the merge does not race structural change to the extractor. Naming:
gotypelint (suffix convention), paired with cuelint on the CUE side.
