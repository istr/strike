Replaces item-0038, which predates several structural sharpenings (the
concept tier, the primitive carve-out, the @go wire/concept split, the
Peer-into-endpoint route) and was fuzzily scoped. This item subsumes 0038's
deploy-target and provenance extraction and widens it to the whole
artifact-centric cluster.

Expanded solution space -- EXPLICIT. The planning model must think wider than
"move files". All of these are permitted, encouraged moves: merge, split,
deduplicate, delete, rename, reorder, combine. Plan the cleanest structure;
do not mechanically relocate CUE files one-to-one. Concretely in play:

- Granularity: one `artifact` package versus several sibling concept
  packages (artifact carrier, artifact/SBOM records, provenance, target,
  digest). ADR-048 section 5 is the binding constraint: it warns a
  meta-named grab-bag reads worse at the call site, and uses `digest.Digest`
  (not `artifact.Digest` or `vocab.Digest`) as the worked example -- which
  pushes toward sibling concept packages, not one bucket. Decide granularity
  against that rule, not by the starting file sketch.
- Dedup: #ArtifactRecord/#SBOMRecord exist twice -- as CUE in
  attest/artifact-record.cue (validation-only, no @go) and as hand-written Go
  structs in internal/deploy/deploy.go. Collapsing to one generated,
  CUE-single-sourced type is a "Meaning is single-sourced" / CUE-first
  (ADR-004) fix, not mere tidiness. Weigh merge-into-carrier vs keep-distinct.
- Deploy destination: a deploy-destination type extracted from lane.cue
  (Ingo's deploy.cue) versus the existing #DeployTarget -- one concept or two.
- Peer: attest's last lane edge besides target/provenance is lane.#Peer
  (over endpoint.#TLS/#SSH). Removing it (route discussed: home #Peer in the
  endpoint concept package, since concept.mayDependOn is [primitive] and a
  separate peer package would need a forbidden intra-concept edge) is what
  lets "attest no longer reaches into lane" be fully, not partially, true.

Overlap to reconcile at planning: item-0039 (collapse DigestRef + wire Digest
into one digest concept) and item-0032 (the digest split it supersedes). This
item lists digest.cue in the cluster, but ADR-048 section 5's own digest.Digest
example argues digest is its OWN sibling concept package -- i.e. item-0039's
internal/digest home. So digest is most likely NOT folded into an artifact
package; either fold item-0039 into this cluster arc as the digest sibling, or
leave digest to item-0039 and have this item reference it. Resolve before
authoring so the digest work is not done twice or contradictorily.

Grounding starting points (drift-invariant, verify at authoring pin):
- #Artifact is an internal-API carrier in lane (artifact.cue); no external
  `lane.Artifact` consumer was found -- confirm whether it is under-used
  before relocating, and scope its move accordingly.
- attest consumes only #DeployTarget from the whole deploy cluster
  (DeploySpec/Method/Recording/ArtifactRef stay lane-only input-wire).
- The ADR-044 concept-tier amendment already exists ("initially
  internal/endpoint"); growing the concept tier needs only a .go-arch-lint.yml
  member change, no ADR edit. No ADR edits are expected for this item; if a
  new tier rule is needed it is an append-only ADR amendment, not an item edit.

The byte-exact instruction is authored ephemerally at execution time against
the then-current pin; nothing byte-exact is stored here. Execution form is
expected to be delegate-measure for the cross-package gengotypes output.
