Drift-invariant notes:

Origin. The item-0034 Identifier cascade (step 1, landed) propagated a named
type through the hand-written attestation and predicate structs in the deploy
package -- the sealed attestation, the most security-sensitive serialization in
the system. That these wire types are hand-written Go rather than generated from
CUE is exactly the kind of gap that pulling type-safety through is meant to
surface: a data schema that lives outside the single source of truth. The
attestation/predicate DTOs are only the exemplar; this item is the exhaustive
sweep the discovery implies.

Principle. Every cross-package or serialized data structure is defined in CUE and
its Go generated; nothing data-shaped is hand-defined. External schemas are not
an exception to this -- they are mirrored explicitly in CUE. The precedent is
the typed CUE replica of the sigstore trusted root: it documents the upstream
spec, tracks a named subset, and confines drift to one file. The in-toto / DSSE
attestation predicate types get the same treatment (a typed CUE replica of the
predicate shape, documenting the upstream framework), not a hand-written Go
mirror.

Detection. The generated Go is gitignored and absent from the tree (it is a
build artifact of the generate step), so the audit cannot find CUE-backed types
by scanning for generated files. It reasons from the CUE source instead: every
CUE type yields a Go type; a hand-written Go data struct with no CUE origin is
the gap. The sweep must be complete -- the value is in leaving no data schema
unaccounted.

Classification. Every hand-written Go type definition is sorted into exactly one
of: (a) data-schema type -- serialized, schema-validated, or carried
cross-package as data -- which is migrated to CUE and its hand-written struct
deleted; (b) disjunction sum-type glue -- the Go interface and concrete-arm
machinery for the @go(-) disjunctions, which CUE declares but whose Go sum-type
representation is necessarily hand-written; (c) behavioral / service type --
holding interface fields, function-typed fields, sync primitives, or OS handles
that CUE structurally cannot express, and carrying no schema identity. The only
pre-approved hand-written category is (b). Category (c) is not a silent
exemption: the audit enumerates it as a closed list for operator ratification, so
nothing data-shaped can hide behind "it is a service object." The end state is
that the only purely hand-written type-definition files are the disjunction glue.

Safety. A named CUE-generated type marshals identically to the hand-written one
(the scalar types carry no custom text/JSON marshaler), so each migration is
wire-neutral and the reproducibility goldens stay byte-identical. This was
confirmed empirically when the item-0034 step-1 cascade retyped these same
attestation fields with zero golden drift.

Sequencing and dependency. The audit/enumeration is the first phase; migration
then proceeds per schema cluster, one cluster per PR. The illustrative
cluster list must be derived from the real hand-written-versus-gengotypes
split at audit time, not assumed from this note: the attestation predicate
and the record/digest-set DTOs it carries are genuinely hand-written and are
the core targets, but output and lane are already in the gengotypes list
(their Go is generated), so output handles are not a migration target -- the
only hand-written piece in internal/output is the disjunction-glue interface,
category (b). The audit confirms each candidate is actually hand-written
before migrating it. Each cluster lands after the type-cleanliness instructions that touch the
same types, so the CUE definition captures the final typed field shape (for
example the Identifier-typed peer maps from step 1) rather than codifying a shape
the type-cleanliness arc then changes. CUE-first policy requires operator
confirmation before each CUE schema change; this item is a gated sequence of such
additions. The byte-exact migration instructions are authored ephemerally at
execution time, never stored here.

Open question carried to ratification. The data-schema-vs-behavioral line is the
single judgment the audit turns on. The behavioral/service set that is kept
hand-written must be enumerated and ratified as a closed list; if the operator
wants that line drawn differently (stricter, or a second pre-approved category),
it is set before ratification.

Retrospective adjustment (dual type-safety audit): the engine option-struct
fields of register row R-02 -- RunOpts.Workdir, Seed.Path, Mount.Target,
VolumeMount.Dest, the ImageVolume fields, and the Tmpfs map key -- are absorbed
into this item's container CUE work, so those structs are touched once rather
than typed now and regenerated later.
