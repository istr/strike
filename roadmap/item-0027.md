Surfaced while wiring item-0001. The podman containerCreate client ignored the
HTTP status entirely: a failed create returned an empty container id, and the
subsequent containerStart hit /containers//start, which the engine answers with
a misleading "405 Method Not Allowed". The real cause (an unresolvable base
image) was invisible. That one client was fixed in place; this item is the
systematic sweep so no other HTTP call site -- in production helpers or in test
doubles -- can mask a wrong status the same way.

Two failure modes to look for: (1) production code that decodes a body without
checking status; (2) mock/stub engines whose hardcoded status diverges from the
real route's status, which lets a too-strict or too-lax production check pass
under test while failing against a real engine (the deploy mock returned 200 for
create where podman returns 201 -- the create check had to accept any 2xx, not a
single code, to stay faithful). Prefer asserting the real route's status in
doubles over relaxing the production check.
