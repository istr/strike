Follow-on to item-0042 (which homed the resolver declaration as endpoint.TLS).
Two contract-level asymmetries remain between the declared resolver and the
observed resolver identity recorded in the sealed attestation:

- The lane declares exactly one DoT resolver (mandatory), and execution gates
  the run on a pre-flight probe that hard-fails (process abort) when the dialed
  and observed TLS identity diverges from the declared trust anchor. So a
  completed run has always verified the resolver -- yet #Sealed marks
  `resolver?` optional. The optional marker contradicts both the mandatory
  declaration and the unconditional probe gate; it should be required.

- The observed record carries `host: string`, structurally disconnected from
  the declared endpoint.TLS it was matched against. The declared->dialed->
  observed match is enforced imperatively at probe time but is not expressed in
  the internal API contract. A typed linkage should connect the declared
  endpoint with the observed connection so the match is contract-visible.

V-property note: a declared-vs-observed trust mismatch is ALREADY a hard
execution error (the pre-flight probe aborts before any attestation is
written). This item does not introduce that guarantee; it makes the contract
express it and adds a regression test so the hard-fail cannot silently
regress. Scope is the contract + internal API + the producer wiring; the verify
path is out of scope.

Open question for execution: whether the typed linkage carries the full
endpoint.TLS or only the representation-neutral endpoint.Address sub-value,
and whether the declared anchor is echoed into sealed or left to the lane file
(laneDigest). Both interact with the sealed wire bytes, so the goldens move and
the change is CUE-schema-gated (stop-and-ask).
